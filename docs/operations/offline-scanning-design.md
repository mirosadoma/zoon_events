# Offline Scanning Design

This document describes the Phase 2 offline-tolerant scanning contract for staff
devices. It satisfies FR-024 and complements `specs/003-wallet-passes-scanning/contracts/scan-contract.md`.

## Allowlist export

`GET /api/v1/tenant/events/{event_id}/offline-allowlist` returns a bounded,
time-windowed allowlist generated by `GenerateOfflineAllowlistAction`:

- `issued_at` — server timestamp when the export was created
- `expires_at` — `issued_at` plus the requested `window_minutes` (15–720) or the
  configured default (`wallet.offline_allowlist_default_window_minutes`, 240)
- `entries` — one object per **active** credential on the event, each containing
  only `credential_reference_digest` (`SHA-256` of the credential id)
- `entries_digest` — `SHA-256` of the sorted, concatenated entry digests

The export never includes a raw QR token, attendee identifier, or contact data.

## Local device dedupe (client contract)

Device software that records scans while offline **must** reject a repeated scan
of a credential that is already stored in the device's local offline journal,
before any server contact. The server does not expose an endpoint for this rule;
it is a client-side requirement to prevent unbounded duplicate submissions.

## Server reconciliation

`POST /api/v1/tenant/events/{event_id}/offline-scan-batches` accepts a
`device_reference` and an array of locally recorded scans (`qr_payload`,
`scanned_at`, optional override fields).

`ReconcileOfflineScanBatchAction`:

1. Creates one `offline_scan_reconciliation_batches` row with `status=received`.
2. Sorts incoming scans by `scanned_at` ascending.
3. Replays each scan through `ScanDecisionEvaluatorImpl` with `offline_mode=true`
   using **current authoritative state** at reconciliation time.
4. Persists each outcome through the same `SubmitScanAction` path used for online
   scans (audited transaction, summary counters, scan events).
5. When two offline-accepted scans of the same credential conflict, the scan with
   the **earliest** `scanned_at` remains `accepted`; later scans become
   `duplicate` with reason `offline_conflict_resolution`. Both owning batches
   increment `conflict_count` and the batch status becomes
   `processed_with_conflicts` when any conflict occurred.

Audit actions: `offline_scan_batch.received`, `offline_scan_batch.processed`,
`offline_scan_batch.conflict_flagged`.
